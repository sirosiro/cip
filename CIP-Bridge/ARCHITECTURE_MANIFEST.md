# CIP-Bridge (Context-Integrity Preservation Bridge) Architecture Manifest

AIエージェントのフラクタル配置を用いた開発工程において、設計者の意図や文脈（Context）が断片化したり損なわれたりすることなく（Integrity）、一貫性を保ったまま（Preservation） にAI群へ伝達・同期させるための接続機構、文脈整合性維持ブリッジ CIP-Bridge です。

## 1. 目的
本ドキュメントは、CIP-Bridge システムの設計意図、コンポーネントの責務、および不変条件を定義する。
AIエージェント間の「意図」を正確に伝達し、ストリーミング出力やターミナル再描画によるノイズを排除した堅牢な通信を保証することを目的とする。

## 2. 実装の核となる原則
- **意図（Intent）の最優先:** 構造よりも「なぜそうしたか」を重視する。
- **正規表現の禁止:** AIは正規表現のエスケープシーケンスに混乱し複雑な条件の生成に失敗します。挙動の不確実性を排除するため、全ての文字列解析において `re` モジュールの使用を一切禁止し、純粋な文字列操作（find, split, replace等）のみで実装する。
- **一回限りの伝達（Once-and-only-once）:** 同一の意味を持つメッセージは、セッション中に二度送信してはならない。
- **自己完結的なコンテキスト:** 各階層は、上位の文脈に依存せず、自身の責務のみを完遂する。

## 3. 主要なアーキテクチャ決定
- **コンテンツベースの重複排除:** 描画の揺らぎ（空白・記号・ANSI残渣）を無視するため、正規化した文字情報のみでハッシュ判定を行う。
- **4階層モデルの採用:** Transport, Protocol, Core, UI の各層を分離し、将来の拡張性を確保する。

## 4. コンポーネント設計仕様

### 4.1. Utils (text_utils.py)
- **責務:** 端末出力からのノイズ除去とストリームのクレンジング。
- **不変条件:** 正規表現 (`re`) を使用せず、ステートマシンまたは単純な文字列走査によって以下のノイズを物理的に濾過する。
    - **ANSIシーケンス:** 色、カーソル制御等の非可読バイナリ。
    - **Thinkingブロック:** AI（Gemini等）の内部思考過程（`<thinking>...</thinking>`）。
    - **CLI装飾:** スピナー（⠋⠙..）、枠線（╭─╮..）、プロンプト残渣などのUIノイズ。 AIマーカー `✦` はパケット境界の識別に用いるため保護を継続。

### 4.2. Transport Layer (fs_bus.py)
- **責務:** ノード間のパケット配送と通知。
- **疎結合設計:** 現状は共有ファイルシステムを用いた `FSBus` を標準実装とするが、`MessageBus` 基底クラスを介した疎結合設計を維持し、将来的に **ネットワーク越し（gRPC/WebSocket等）のトランスポートへの挿げ替え** を容易にする。
- **トポロジー自動発見:** 各ノードは自身の起動ディレクトリから最大10階層上まで `cip_bus` を自動探索し、シンボリックリンクを生成することで、フラクタルな階層構造における通信路を自律的に確立する。
- **通知メカニズム:** メッセージの書き込み後、対象ノードのPIDに対して `SIGUSR1` を送信し、受信側のイベントループを即座にキックする。

### 4.3. Node Lifecycle & Reliability
- **PID管理:** 各ノードは一意の `bus_id` に紐付いた `bridge.pid` を発行し、二重起動を厳格に防止する。異常終了したノードの残骸は次回の起動時に自動クリーンアップされる。
- **ツール実行の自動承認:** AIが `read_file` 等のツール実行許可を求めている状態（"Allow execution of:"）を検知した場合、Bridgeが人間に代わって `ENTER` を自動送信し、自律的な作業継続をサポートする。

### 4.4. ProtocolStack (stack.py)
- **責務:** テキストからのタグ抽出、正規化、およびエコーバック防止。
- **不変条件:** 同一の「意味内容（正規化されたテキスト）」を持つパケットは、Bridgeの生存期間中、二度と抽出してはならない（Content-based De-duplication）。
- **宛先抽出規則:** [TAG] の直後（先頭の空白・改行を除く）に続く `@ID` のみを正当な宛先指示として認め、それ以外の本文中の言及は無視する。

### 4.5. NegotiationManager (negotiator.py)
- **責務:** ルーティングと最終重複防御。
- **最終防衛線:** 送信直前（`should_route`）において、過去に送信した全パケットの正規化ハッシュ履歴（最大100件）と照合し、重複を物理的に遮断する。

## 5. 通信プロトコル仕様 (Protocol Specification)
AIエージェント間で交換されるメッセージは、以下のタグで構造化される。
- **`[SYSTEM]`**: システム（人間またはリーダー）からの権威ある指示。
- **`[NEED_CONSENSUS]`**: 他ノードへの提案または合意形成の要求。
- **`[ACCEPTED]` / `[CONFLICT]`**: 要求に対する受諾または対案。
- **`[COMPLETED]` / `[FAILED]`**: タスクの完了または異常終了（交渉の打ち切り）。

## 6. フラクタル・ネゴシエーション・ポリシー (Fractal Negotiation Policy)
- **直列対話の強制:** コンテキストの汚染を防ぐため、一度に複数の Worker へ `[NEED_CONSENSUS]` を発行することを禁止する。必ず一対一の合意形成を完遂させてから次へ進む。
- **有限回数の調整:** 合意に至るまでの往復回数は最大3回までに制限し、解決不能な矛盾は速やかに上位（人間）へエスカレーションする。
