  CIP-Bridge 未実装機能・アイデア 調査レポート（有用性順）

  🥇 1位：@cip トリガーと「動作モード (Bypass / Auto)」の動的切り替え
   * 仕様書の記載: CIP-Bridge_spec.md (2.1, 3章)
   * 概要: ユーザー入力が @cip で始まった際に「Auto
     Mode（ユーザー入力を遮断し、バックグラウンドでのAI間交渉に専念するモード）」へ遷移し、それ以外は「Bypass
     Mode（通常のGemini CLIとして振る舞う）」とする機能。同時に、@cip
     起動時に稼働中のWorkerリスト（.pid）をスキャンし、Leaderに動的に注入する。
   * 現状: 未実装。bridge.py では sys.stdin を常にパススルーしており、モードの概念自体が存在しません。
   * 有用性の理由【最高】:
     人間の操作とAIの自律交渉がターミナル上で衝突するのを防ぐための根本的な機能であり、Bridgeが単なる「土管」から「オー
     ケストレーター」に進化するために不可欠です。

  🥈 2位：不在Worker指定時の「システムフォールバック通知」
   * 仕様書の記載: CIP-Bridge_spec.md (5.2章)
   * 概要: Leader AI が [NEED_CONSENSUS] @cpu_emu
     のように存在しない（起動していない）Workerを指定した場合、Bridgeがそれを検知し、LeaderのPTYに対して即座に「Worker不
     在」のシステムメッセージを打ち返す機能。
   * 現状: 未実装。現状の FSBus.send は宛先がない場合に False を返し、内部ログに書き込むだけで、Leader
     AIには何も通知されません。
   * 有用性の理由【高】: DESIGN_PHILOSOPHY.md で定義されている
     「専門家不在時はLeader自身が兼任して決断する」という最重要ルールのトリガー
     となる機能です。これが無いと、AIは返事を永遠に待ち続けてしまいます。

  🥉 3位：[COMPLETED] タグによる合意完了ロジックと解決フェーズ
   * 仕様書の記載: CIP-Bridge_spec.md (3章, 4章)
   * 概要: 全ての [ACCEPTED] を確認した後、Leader AI が [COMPLETED] を出力し、Bridgeがそれを検知して「Auto
     Mode」から「Bypass Mode」へ確実に戻す（人間に制御を返す）仕組み。
   * 現状: 未実装。ProtocolStack クラスの抽出対象タグは NEED_CONSENSUS, ACCEPTED, CONFLICT
     の3つのみであり、終了宣言をハンドリングしていません。
   * 有用性の理由【高】:
     現在の交渉ループは「AIが喋らなくなったらなんとなく終わる」という不安定な状態です。明確な終了状態（State）を持つこと
     でシステムが堅牢になります。

  4位：Transaction ID の導入による交渉状態（State）の厳密な管理
   * 設計書の記載: CIP_BRIDGE_REFACTOR_PLAN.md (5章)
   * 概要: [NEED_CONSENSUS] の発生ごとに一意のトランザクションIDを付与し、その後の [ACCEPTED] や [CONFLICT]
     をそのIDに紐付けて管理する仕組み。
   * 現状: 未実装。現在の NegotiationManager
     は、直近の「メッセージ内容のハッシュ値履歴」だけでエコーバックや無限ループを強引に防いでいる状態（ハック）です。
   * 有用性の理由【中】:
     複雑な反論（CONFLICT）が複数回続くような高度な交渉になった場合、現在のハッシュベースの重複排除では意図しない通信の
     遮断が起きるリスクがあるため、将来的に必要になります。

  5位：RedisBusとブロードキャスト（ID: ALL / MAGI通信）
   * 設計書の記載: CIP_BRIDGE-BUS-EXTENSION.md, CIP_BRIDGE_REFACTOR_PLAN.md
   * 概要:
     FS-Bus（ファイルシステム）を抽象化した上で、RedisのPub/Sub機能を用いたネットワーク越しの分散ワーカートランスポート
     や、@ALL 宛に送信することで複数AIと同時に交渉する「MAGIシステム」の実現。
   * 現状: 未実装。BaseBus による抽象化の片鱗はありますが、実装はローカルディレクトリ型の FSBus
     と1対1のルーティングに留まっています。
   * 有用性の理由【中】:
     ロマンと拡張性は非常に高いですが、まずは1台のPC内での1対1の自律交渉（上記1〜3位）を完璧に動作させることが先決である
     ため、優先度は少し下がります。

  に組み込むことが、設計思想を具現化するための最短ルートと考えられます。
