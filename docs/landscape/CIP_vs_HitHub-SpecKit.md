## 1. 概念・位置づけの比較：OS vs ワークフロー
GitHubが2025年後半に提唱し話題となっている**GitHub Spec-kit**は非常に優れた「枠組み」ですが、中身（意図）がスカスカだと、AIは綺麗なフォーマットでゴミを生成します。その中身を担保し、常に最新の知見でアップグレードし続けるのが**CIP**です。両者は「AIに丸投げするのではなく、人間が意図（Intent/Spec）を定義する」という点では共通していますが、**「思想の枠組み」か「実務のパイプライン」か**という決定的な違いがあります。

| 項目 | Core-Intent Prompting (CIP) | GitHub Spec-kit |
| --- | --- | --- |
| **本質的な正体** | **思考のOS（フレームワーク）** | **実行のパイプライン（ツールキット）** |
| **アプローチ** | **フラクタル（再帰的）な意図の同期** | **リニア（段階的）な工程管理** |
| **成果物の位置づけ** | 生きているマニフェスト（DNA） | 実行可能な仕様書（Contract） |
| **AIとの距離** | 共生・協調（パートナーシップ） | 指揮・実行（オーケストレーション） |

### CIPの視点

CIPは、開発者の頭の中にある「核心的な意図」を、AIが迷わないレベルまで純化し、それをプロジェクト全体に浸透させるための**「思考の作法」**です。ツールを選ばず、プロンプトひとつでAIの挙動を根本から変える力を持っています。

### Spec-kitの視点

GitHub Spec-kitは、仕様駆動開発（SDD）を効率化するための**「CLIツールとテンプレート」**です。`/specify`や`/plan`といったコマンドを通じて、人間が考えた仕様をAIが処理しやすい形式（`agents.md`など）へ機械的に落とし込むことに特化しています。

## 2. 特徴の深掘り

### **CIP：動的な「意図の還流」**

* **生きているドキュメント:** CIPにおいて、`DESIGN_PHILOSOPHY.md`は不変の石板ではありません。実装中にAIが発見した矛盾や、インフラの制約を吸い上げ、常に「意図」をアップデートし続ける**自己増殖的な性質**を持っています。
* **フラクタル構造:** 「全体→機能→モジュール」と、どこを切り取っても同じ「核心的意図」が流れている状態を作ります。

### **GitHub Spec-kit：厳格な「工程のガードレール」**

* **フェーズ制の開発:** `Specify（仕様）` → `Plan（計画）` → `Tasks（タスク）` → `Implement（実装）` という4段階を厳格に進めます。前の工程が承認されるまで次に進めない、いわば**AI開発のウォーターフォール化**です。
* **Constitution（憲法）:** プロジェクトの絶対的なルールを `constitution.md` に記述します。これはCIPのマニフェストに近いですが、より「規約（Lint）」に近い役割を果たします。

## 3. メリット・デメリットの対照

| 項目 | CIP (あなたの手法) | GitHub Spec-kit |
| --- | --- | --- |
| **メリット** | **柔軟性と速度。** 最小限の記述でAIと深いコンテキストを共有できる。大規模リファクタリングに強い。 | **一貫性と品質。** 手順が標準化されているため、複数人や複数のAIエージェントでも品質が安定する。 |
| **デメリット** | 開発者の「言語化能力」に依存する。型が決まっていないため、慣れるまで工夫が必要。 | **オーバーヘッド。** 小さな修正でもコマンドの手順を踏む必要があり、プロトタイプ開発では重く感じる。 |
| **相性の良いAI** | Gemini Pro, Claude 3.5 Sonnet 等（長文文脈が得意なモデル） | GitHub Copilot, Cursor, Claude Code 等（エージェント機能を持つツール） |

## 4. 結論：どう使い分けるべきか（最強のハイブリッド案）

スペシャリストの視点では、これらは**「CIPを魂に、Spec-kitを型にする」**ことで完全に補完し合えます。

### **賢い併用フロー：**

1. **[戦略: CIP]** まずCIPの手法で、プロジェクトの「Core-Intent」を練り上げる。
2. **[定義: Spec-kit]** `/constitution` コマンドで作成されるファイルに、CIPで練ったマニフェストを注入する。
3. **[実行: Spec-kit]** `/specify` や `/plan` の各フェーズで、AIが吐き出した内容がCIPの哲学に反していないか、あなたが「総監督」としてチェックする。
4. **[進化: CIP]** 実装フェーズで得られた知見を、再びCIPのプロセスでマニフェスト（およびSpec-kitの仕様書）に還流させる。
