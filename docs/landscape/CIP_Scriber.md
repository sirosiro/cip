# CIP-Scriber 言語仕様書 (Version 1.5)

## 1. 言語の核心的意図 (Core Philosophy)

CIP-Scriberは、人間とAIが「意図」を共有し、実コード（Python, C++, Rust等）へ安全に橋渡しするための中間言語である。

* **レビュー負荷の最小化**: 「どう実装したか」ではなく「何を意図したか（@理由）」と「論理フロー（->, =>）」の整合性をチェックするだけで検証可能とする。
* **OSSリスクの物理一掃**: `policy` ブロックにより、AIによる無秩序なライブラリ導入を構造的に遮断する。
* **AIサボり防止**: 日本語の「文末確定性（SOV型）」を機能説明に用いることで、AIに論理の完結を促す。
* **意図による動的実装**: 複雑なOS制御や定型処理は、アノテーションによる「概念の定義」としてWorker AIに委託する。

## 2. 構文規則 (Syntax Rules)

### 2.1 識別子の保護

* **日本語識別子**: 変数名、関数名、構造体名、定数名は必ず **バッククォート ( ` )** で囲む。
* **ASCII命令**: 演算子、数値、基本型名は半角ASCII文字を使用する。

### 2.2 データ構造の定義 (Data Structure)

* **構造体**: `構造体名` :: { `フィールド名`: 型, ... }
* 例1: ``レジスタ` :: { `A`: 8bit, `F`: 8bit }`
* 例2: ``ユーザー` :: { `ID`: 数値, `名前`: 文字列 }`


* **定数**: ``定数名` :: 値`
* **初期化指示**: `@初期値:` アノテーションを用いてデフォルト状態を定義する。
* 例: ``最大再試行` :: 3`

### 2.3 命令とフロー

* **`::`** : 定義（名前と内容の結合）
* **`->`** : データの流動。代入、パイプライン的な連続変換、出力を兼ねる。
* **`=>`** : 条件分岐の実行（マッチング）。

## 3. アノテーションによる概念の想起 (Conceptual Annotations)

* **`@理由:`**: 論理の根拠（レビューの主眼点）。
* **`@実行:`**: 副作用（IO、画面出力等）を伴うアクションの説明。
* **`@役割:`**: 実行性質、または継承・遵守すべきインターフェースの定義。
* **`@制約:`**: アトミック性、永続化、リソース解放（cleanup）等の物理的ルール。
* **`@戦略:`**: アルゴリズム指針、または外部仕様（RFC/マニュアル等）の参照指示。
* **`@例外:`**: 失敗時の振る舞い（エラーハンドリング、バブリング）。

---

## 4. 運用の目的とセキュリティモデル (Review & Security)

### 4.1 エンジニア・レビューの最適化

実コードでは言語固有の装飾に本質が埋没する。Scriberはこれらをアノテーションに分離することで、**「論理の不整合」だけを浮き彫りにする。**

### 4.2 OSS汚染防止（隔離フィルター）

AIが利便性のために外部の未承認ライブラリを導入することを構造的に遮断する。

* **ホワイトリスト制**: `policy` ブロックで許可されたモジュール以外への依存を一切禁止。
* **論理の強制**: ライブラリ丸投げを許さず、Scriber上で論理を書き下させることで、生成コードの「純度」を確保。

---

## 5. 抽象化粒度の決定基準 (Abstraction Levels)

| 記述レベル | 記述範囲 | 推奨されるアノテーション |
| --- | --- | --- |
| **高位 (完全委譲)** | 業界標準（Z80, HTTP, AES等） | `@戦略` に参照規格を記述。 |
| **中位 (論理固定)** | 独自の業務ルール、交渉ロジック | `=>` で条件と結果の関係を明文化。 |
| **低位 (詳細指示)** | 物理層、アトミック性、リソース解放 | `@制約` や `@例外` で厳格な手順を記述。 |

## 6. 付録：完全サンプル (Appendix: Reference Examples)

### サンプル A：データ構造と低レイヤー制御

```cip
`通信パケット` :: {
    `ヘッダ`: 数値,
    `ペイロード`: 文字列,
    `整合性チェック`: 数値
}

`安全な送信` :: (`データ`: `通信パケット`) {
    `データ`.`ペイロード`.`長さ` > 1024 => {
        return `エラー: サイズ超過`
    }

    `ソケット` を開く -> `ハンドル`
    `データ` -> `ハンドル` へ書き込み
    
    @制約: ネットワーク切断時や終了時は、必ず `ハンドル` を閉じ、未送信バッファを破棄すること。
    @実行: 物理デバイスへパケットをアトミックに送出する。
    
    policy :: {
        allow: `socket`, `ssl`
        restrict: `外部ライブラリ` == 禁止
    }
}

```

### サンプル B：状態保持とCPUエミュレーション（Z80抜粋）

```cip
`Z80レジスタ` :: {
    `A`: 8bit, `F`: 8bit,
    `PC`: 16bit @初期値: 0x0000
}

`加算命令` :: (`値`) {
    `A` + `値` -> `暫定結果`
    
    `暫定結果` == 0 => { `F`.`ゼロフラグ` をセット }
    `暫定結果` -> `A`
    
    @理由: Z80の演算仕様に基づき、演算結果をレジスタとフラグに反映する。
    @戦略: ハーフキャリー等の詳細な判定式は、Z80ユーザマニュアルの定義に従いWorkerが補完すること。
}

```

### サンプル C：低レイヤー制御（アトミック書き込み）

```cip
`アトミック書き込み` :: (`ファイルパス`, `データ`) {
    `ロック取得`(`ファイルパス`) => {
        `ファイルパス` -> `一時ファイル作成` -> `ハンドル`
        `データ` -> `ハンドル` に書き込み
        
        `ハンドル`.`同期`()
        `一時ファイル` -> `正式パスへリネーム`
        @実行: リネーム操作を用いて物理的な書き込み完了を保証する。
    }
    
    @制約: 終了時には成否を問わず必ず `ロック` を解放すること（finally）。
    @例外: 失敗時は一時ファイルを削除し、エラーを呼び出し元へ投げること。
    
    policy :: { allow: `os`, `fcntl` }
}

```

### サンプル D：高次論理（CPUサイクルとパイプライン）

```cip
`CPUサイクル` :: () {
    `PC` -> `メモリ読込` -> `デコード` -> `実行` -> `PC更新`
    
    @戦略: 各ステージの処理は、外部仕様 `Z80マニュアル` の定義に準拠すること。
    @理由: フェッチから実行までの一連のフローを一本の変換パイプラインとして定義。
}

```

## 7. 予約語・記号一覧

| 記号 | 名称 | 用途 |
| --- | --- | --- |
| `::` | 定義 | 構造体、関数、定数の定義 |
| `->` | 流動 | データの移動・変換（パイプライン） |
| `=>` | 条件実行 | 条件分岐、マッチングの実行 |
| ``...`` | 識別子 | 日本語（ASCII外文字）の保護 |
| `@理由` | 理由 | なぜその論理なのか（レビュー重要項目） |
| `@初期値` | 初期値 | データ構造の初期状態の定義 |
| `policy` | ポリシー | 外部機能の承認・制限（OSS混入防止） |
