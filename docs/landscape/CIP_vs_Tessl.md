## AI駆動開発比較レポート：CIP vs Tessl
2025年後半から急速に注目を集めている**Tessl**は「仕様がすべて」という極端なアプローチを採っていますが、だからこそ**「仕様そのものをどう進化させるか」というCIPの視点**が不可欠になります。
Tesslという「工場」に、CIPという「設計局」を組み合わせる。この二つが揃って初めて、あなたは実装という重労働から真に解放され、純粋な「設計」と「意図」の領域で開発を楽しむことができるようになります。

Tesslは「コードではなく仕様を正本とする」という、これまでの開発の常識を覆すプラットフォームです。CIPとTesslは、**「意図（Intent）を最優先する」**という哲学において極めて高い親和性を持っています。

### 1. 概念・位置づけの比較

Tesslが「仕様を管理・実行するインフラ」であるのに対し、CIPはその仕様の中身を「どう定義し、どう進化させるか」という知能の部分を担います。

| 項目 | Core-Intent Prompting (CIP) | Tessl |
| --- | --- | --- |
| **定義** | **意図の構造化メソッド（OS）** | **AIネイティブ仕様駆動開発プラットフォーム** |
| **キーワード** | **Intent（意図）・Living Manifest** | **SDD (Spec-Driven Development)** |
| **役割** | 人間の設計思想をAIに同期・進化させる | 「仕様」からコードを自動生成・保守する |
| **成果物の正本** | **進化し続けるマニフェスト** | **AIが解釈可能な「仕様（Spec）」** |

---

### 2. 特徴の深掘り

#### **Tessl: 「コードを書かない」という究極の自動化**

Tesslは、開発者が「コード」を直接編集することをやめ、「仕様」のみを更新し続ける世界を目指しています。

* **Spec as Source of Truth:** 仕様書が唯一の正解であり、コードはそこから生成される一時的な副産物に過ぎないと定義します。
* **自動メンテナンス:** 言語のアップデートやライブラリの変更も、仕様が変わらなければTessl側が自動でコードを再生成して対応します。
* **2026年の強み:** 特定のプログラミング言語に依存せず、仕様から複数の言語への同時展開も可能です。

#### **CIP: 「仕様」の魂を定義し、進化の方向を決める**

Tesslが「仕様をコードに変えるエンジン」なら、CIPはその仕様に**「どのような魂（意図）を込めるか」**を司ります。

* **意図の純化:** Tesslに渡す「仕様」が曖昧だと、生成されるシステムも使い勝手の悪いものになります。CIPは、`DESIGN_PHILOSOPHY.md`を通じて、仕様の背後にある「真の狙い」を言語化します。
* **フィードバックの還流:** Tesslで生成・テストした結果得られた「動作上の限界」や「ユーザー体験の不備」を、即座にマニフェストへ戻し、仕様をアップデートする**「動的な設計」**を可能にします。

---

### 3. メリット・デメリットの対照

| 項目 | CIP (あなたの手法) | Tessl (プラットフォーム) |
| --- | --- | --- |
| **メリット** | **汎用性と知性の進化。** どんなツール（Tessl含む）を使っても、設計の質と一貫性が保たれる。 | **メンテナンスからの解放。** 仕様さえ守れば、実装の詳細（コード）に悩まされることがなくなる。 |
| **デメリット** | 常に「意図」を高い解像度で言語化し続ける思考の労力が必要。 | ツール独自の仕様記述形式に慣れる必要がある。レガシーコードとの統合が難しい。 |
| **相乗効果** | **CIPで「生きている設計思想」を練り、Tesslをその思想を具現化する「自動工場」として使う。** |  |

---

### 4. 結論：どう使い分けるべきか（最強のシナリオ）

2026年、最も高度な開発スタイルは、**「CIPでプロジェクトのDNAを書き、Tesslでその肉体を生成させる」**という形です。

#### **推奨ワークフロー**

1. **[フェーズ1: CIP]**
Gemini Proなどの強力なLLMと対話し、`DESIGN_PHILOSOPHY.md`を構築。システムの「核心的な意図」と「絶対に譲れない価値観」を固める。
2. **[フェーズ2: Tesslへの注入]**
CIPで固めた意図を、Tesslが解釈可能な「仕様書（Spec）」に翻訳して入力する。
3. **[フェーズ3: 自動生成と検証]**
Tesslがコードを生成。あなたはコードを読まず、期待した「意図（Intent）」通りに動いているかを検証する。
4. **[フェーズ4: 進化的フィードバック]**
動作結果を見て「意図と違う」と感じたら、コードを直すのではなく、**CIPのマニフェストを修正**する。修正されたマニフェストから新たな仕様をTesslに流し込み、システムを「進化」させる。
