## 1. 概要 (Overview)

本ドキュメントは、CIP (Context, Intent, and Code Platform*) プロジェクトの長期的なロードマップとして、**「フラクタル配置されたAIエージェント群による自律分散開発エコシステム」**の構想を定義します。

従来の「単一の巨大なAI」が全てを処理するアプローチではなく、システムの構造（ディレクトリ/モジュール）に合わせてAIエージェント（CIPノード）を再帰的（フラクタル）に配置し、それらが**「組織」として協調・交渉することで、大規模システムの開発を可能にするアーキテクチャ**です。

*(Note: CIPの具体的な定義はプロジェクトの文脈に依存しますが、ここでは「コードと意図を管理する自律単位」として扱います)*

---

## 2. アーキテクチャ構造 (Architectural Structure)

このシステムは、人間の組織図に似た階層構造を持ちますが、各ノードは「CIPセット（マニフェスト＋コード＋専任AI）」として完結しています。

### 2.1. 役割定義

*   **Human Engineer (Client)**
    *   システムの最終責任者。
    *   抽象的な要望、ビジネス要件、または特定の問題をCIPリーダーに投げます。

*   **CIP Leader (Orchestrator / Top-Level AI)**
    *   **責務**: 人間とのインターフェース、システム全体の整合性維持、タスクの分解と割り振り。
    *   **能力**: 全体アーキテクチャ（ルート・マニフェスト）を理解していますが、詳細な実装（How）は知りません。
    *   **権限**: 下層CIPへの要求発行、調整案の作成。

*   **Sub-CIP Sets (Specialized Agents / Fractal Nodes)**
    *   **責務**: 特定のモジュール（ディレクトリ）の守護者。
    *   **能力**: 自身の担当領域（ローカル・マニフェストとコード）について深い知識と文脈（コンテキスト）を持ちます。
    *   **権限**: **「拒否権 (Veto Power)」**。上層からの要求が自身の管理領域の整合性を破壊する場合、理由を添えて拒否できます。

## 3. 協調プロセス：ネゴシエーション・ループ (The Negotiation Loop)

このアーキテクチャの核心は、一方的な「命令」ではなく、双方向の**「交渉（Negotiation）」**によって解決策を導き出す点にあります。

### Step 1: 要求の受領と分解
1.  **Human** が **CIP Leader** に要望を出す（例：「ユーザー認証に2要素認証を追加したい」）。
2.  **CIP Leader** はシステム全体への影響を分析し、タスクを分解して関連する **Sub-CIPs**（例：UI担当、DB担当、Auth担当）へ「変更要求（Draft Proposal）」を投げます。

### Step 2: 局所的な吟味と拒否 (Validation & Veto)
3.  各 **Sub-CIP** は、受け取った要求を自身の管理する `ARCHITECTURE_MANIFEST.md` および既存コードと照らし合わせます。
4.  **判定**:
    *   **Accept**: 整合性が取れる場合、承認します。
    *   **Reject**: 問題がある場合（例：「その変更は既存のセッション管理と矛盾し、デッドロックのリスクがある」）、**具体的な拒否理由**を添えてリーダーに返します。

### Step 3: 解決策の再構築 (Synthesis)
5.  **CIP Leader** は、Sub-CIPsからの拒否理由を集約し、システム全体レベルでのトレードオフを再検討します。
6.  必要であれば、代替案を作成したり、他のSub-CIPへの要求を調整したりして、再度提案（Revised Proposal）を投げ直します。

### Step 4: コンセンサスと回答
7.  全ての関連Sub-CIPとの合意（Consensus）が取れた段階で、変更計画が確定します。
8.  **CIP Leader** は整理された解決策と実装計画を **Human** に回答します。

## 4. このアーキテクチャの利点 (Advantages)

### A. トークン枯渇問題の構造的解決
*   単一のAIが全コードを読み込む必要がありません。各AIは自身の担当領域（ディレクトリ）のコンテキストのみを保持すれば良いため、トークン消費を劇的に抑えつつ、深い推論が可能になります。

### B. 品質の防衛線（拒否権）
*   下層AIが「イエスマン」にならず、専門家の立場から拒否権を行使することで、トップダウンの無茶な変更によるシステム破壊を防ぎます。

### C. 安定した考察 (Cognitive Stability)
*   AIを一箇所（特定のCIPセット）に常駐させることで、その領域に関する「記憶」や「学習」が安定し、一貫性のある保守が可能になります。

## 6. 技術仕様案: CIP Communication Protocol (CCP)

本セクションでは、複数のAIエージェント（CIPノード）が協調するための通信仕様と、それを支える実行基盤について記述します。

### 6.1. 通信アーキテクチャ: 専用オーケストレーターとファイルベースIPC

`gemini-cli` は対話型ツールであり、APIを持たないため、Unixパイプや `expect` コマンドのような手法で制御する必要があります。
しかし、標準的な `expect` コマンドでは以下の課題があることが判明しています。
*   **表示崩れ:** エスケープシーケンスや日本語を含む出力の制御が困難。
*   **危険な操作の自動承認:** ファイル書き込み等の重要な確認ダイアログを、文脈に応じて安全かつ確実に自動承認（または拒否）するロジックの実装が困難。

したがって、我々は `gemini-cli` に特化した**専用の制御ツール（CIP Orchestrator / "Super-Expect"）**を開発する必要があります。

#### **CIP Orchestrator の要件**
1.  **堅牢な対話制御:** `gemini-cli` のプロンプト待ち受け、出力解析、入力注入を安定して行う。
2.  **コンテキスト注入:** ファイルシステム上の「リクエストファイル」を読み取り、それをプロンプトとして `gemini-cli` に渡す。
3.  **安全な自動承認ロジック:** ツール実行時の確認プロンプト（[Y/n]等）に対し、あらかじめ定義されたポリシー（例：テスト環境ならY、本番環境ならn）に基づいて応答する。

### 6.2. ディレクトリ構造とメッセージング

各CIPモジュールは、以下のディレクトリ構造を通じてメッセージを交換します。

```text
project/root/
├── docs/cip_communication/
│   ├── active_requests/     # 進行中の交渉（Inbox）
│   │   ├── REQ_20260215_001_AuthUpdate/
│   │   │   ├── 00_LEADER_PROPOSAL.md   # リーダーからの指令書
│   │   │   ├── 01_SUB_AUTH_REPLY.md    # Auth担当からの返信（拒否/承認）
│   │   │   ├── 02_SUB_UI_REPLY.md      # UI担当からの返信
│   │   │   └── 03_LEADER_VERDICT.md    # 最終決定
│   └── archived_requests/   # 完了した交渉ログ
```

### 6.3. 透明性の担保 (Transparency Policy)

*   **Open Access:** 全てのCIPエージェントは、他者のメッセージおよび内部状態（マニフェスト等）を閲覧可能とする。情報の隠蔽（Encapsulation）は行わず、必要に応じて参照できる透明性を維持する。
*   **Context Linking:** メッセージ内では、判断の根拠となるファイルや行番号への明確なリンク（参照）を提示し、受信者が効率的にコンテキストをロードできるようにする。
