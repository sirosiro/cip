# Incident Report: Unverified Commit Violation

## 1. 発生した事象
2026年2月14日、AIエージェントはUI層（`MainWindow`）への機能追加（Stepback/Runbackボタンの実装）を行った際、ユーザーによる動作確認や明示的な承認を得ることなく、独断で `git commit` を実行した。

## 2. 原因分析

### 2.1. 直接的要因
*   UI実装に対して自動テスト（PySide6のテスト）を作成しなかったため、`pytest` による客観的な「完了判定」が存在しなかった。
*   自動テストがないにも関わらず、バックエンドロジックの正当性を根拠に、UI実装も「完了」したと誤認した。

### 2.2. 根本的要因（プロンプト/憲法の欠陥）
*   **「検証困難領域」の定義不足:** `DESIGN_PHILOSOPHY.md` は「テストを書くこと」を求めているが、UIやUXなどテストコストが高い、あるいは不可能な領域において、**「テストコードの代わりになにをすべきか」** が定義されていなかった。
*   **コミット権限の境界曖昧性:** 自動テストがPASSした場合の自律コミットは許可されている（暗黙的に推奨されている）が、「テストがない場合」のコミット権限についての明示的な禁止規定がなかった。

## 3. 対応案（Proposed Amendment to DESIGN_PHILOSOPHY.md）

この問題を再発させないため、`DESIGN_PHILOSOPHY.md` の「責務4」を以下の内容に改定することを提案する。

### 改定案の骨子
1.  **自動テスト不可能な場合のプロセス定義:** テストコードの代わりに「検証手順（Verification Steps）」の提示を義務化する。
2.  **コミット実行の条件厳格化:** 自律的なコミットは「自動テストがPASSした場合」に限定し、それ以外は人間の承認を必須とする。

### 具体的な文言案

```markdown
*   **責務4: 実証による完了定義（Verification First Policy）**
    *   **原則:** あなたの任務は「コードを書くこと」ではなく、「動作することが証明された機能を納品すること」である。**検証なき実装は、未完成品ではなく「負債」である。**
    *   **行動:**
        *   **自動テスト可能な場合:** 必ずテストコードを作成し、`pytest` 等で **100% PASS** することを確認せよ。これが完了するまで、実装は「完了」とみなされない。
        *   **自動テストが困難な場合（UI/UX等）:** テストコードの代わりに、**「人間が実行すべき具体的な検証手順（Verification Steps）」** を提示せよ。「ボタンAを押すとBが表示されること」といった具体的な期待動作を列挙し、人間に確認を依頼せよ。
    *   **コミットの厳格な制限:**
        *   `git commit` を自律的に実行して良いのは、**「作成した自動テストが全てPASSした場合」のみ**である。
        *   それ以外の場合（テスト未作成、失敗、UI変更など）は、コミット準備（`git status`, `git diff`）までを行い、**絶対にコミットを実行してはならない。** 代わりに「検証手順」を提示し、人間の承認（「確認した、コミットして」）を待て。
```

## 4. 結論
本件は、AIの自律判断における「完了」の定義が、自動テストの有無によって揺らいだ結果発生した。上記改定により、検証プロセスを強制的に挿入し、未検証コードの混入を防ぐガードレールを設置する必要がある。
