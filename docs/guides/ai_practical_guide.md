# AI駆動開発への道：『AIエージェント』との協調

## はじめに

このドキュメントは、[`ai_usage_guidelines.md`](./ai_usage_guidelines.md)で解説された「AIを正しく使うための基礎体力と思考法」を習得したエンジニアが、次のステップへ進むための応用・実践編です。

ここでのゴールは、AIを単なる「便利な道具」として使うレベルから、**システムの設計思想を理解し、自律的に提案や実装を行える「設計パートナー（AIエージェント）」として協調する**レベル（[Google検索：「AI駆動開発 Level 3」](https://www.google.com/search?q=AI%E9%A7%86%E5%8B%95%E9%96%8B%E7%99%BA+Level+3)）に到達することです。

`ai_usage_guidelines.md`であなたが「優秀な機長」になった今、次はその「副操縦士」を「真のパートナー」へと育てる旅を始めましょう。

---

## 学習の進め方

このドキュメントでは、架空の小プロジェクト「**シンプルなコマンドラインタスク管理ツール**」を題材に、以下のステップでAIとの協調レベルを段階的に引き上げていきます。

1.  **Step 1: プロジェクトの全体設計をAIに伝える（グローバル指示書の作成）**
2.  **Step 2: コードに「設計意図」を埋め込む（インテント・コメントの実践）**
3.  **Step 3: AIエージェントとの対話的開発**

## Step 1: プロジェクトの全体設計をAIに伝える（グローバル指示書の作成）

大規模プロジェクトでは、個々のコードだけでなく、プロジェクト全体を貫く設計原則やアーキテクチャ上の制約が存在します。これらをAIに最初に伝えるための「グローバル指示書」（アーキテクチャ・マニフェスト）を作成します。

これにより、AIは場当たり的な提案ではなく、プロジェクト全体の一貫性を保つ提案をしてくれるようになります。

### 雛形（テンプレート）

以下は、我々の架空プロジェクトで使うグローバル指示書の雛形です。実際のプロジェクトでは、これを元にカスタマイズして運用します。

```markdown
# Architecture Manifest for "Simple Task Manager"

## 1. プロジェクトの目的 (Purpose)

このプロジェクトは、コマンドラインで完結するシンプルなタスク管理ツールを開発することを目的とする。ユーザーはタスクの追加、一覧表示、完了、削除ができる。

## 2. 技術スタック (Tech Stack)

-   **言語**: Python 3.10+
-   **外部ライブラリ**: なし（標準ライブラリのみで完結させる）

## 3. 設計原則 (Design Principles)

-   **P1: 単一責任の原則 (Single Responsibility Principle)**: 各モジュール、各クラス、各関数は、単一の責任のみを持つこと。
-   **P2: シンプルさ (Simplicity)**: 複雑なデザインパターンよりも、コードの可読性とシンプルさを優先する。
-   **P3: 状態の分離 (State Separation)**: タスクのデータ（状態）を管理するロジックと、ユーザーインターフェース（表示）のロジックを明確に分離する。

## 4. アーキテクチャ上の制約 (Architectural Constraints)

-   **AC1: ファイルアクセス**: タスクデータは単一のJSONファイル（`tasks.json`）に保存する。データアクセスを行うモジュールは`data_manager.py`のみに限定する。他のモジュールが直接ファイルを読み書きしてはならない。
-   **AC2: エラーハンドリング**: 想定外のエラーが発生した場合でも、データファイル（`tasks.json`）が破損しないことを最優先とする。具体的には、書き込みは一時ファイルに対して行い、成功した場合のみリネームするアトミックな操作を徹底する。

## 5. 将来の拡張性に関する注記 (Notes on Future Extensibility)

-   現時点では、タスクに優先度や期限の概念は含めない。ただし、将来的にこれらの属性が追加される可能性を考慮し、データ構造は柔軟な辞書のリストとしておく。
```

**【演習】**: あなたが新しいプロジェクトを始めるなら、この雛形の他にどんな項目を追加したいですか？ AIに「ソフトウェア開発プロジェクトのアーキテクチャ設計で考慮すべき項目を10個リストアップして」と聞いてみましょう。

## Step 2: コードに「設計意図」を埋め込む（インテント・コメントの実践）

グローバル指示書が「森」の地図なら、インテント・コメントは「木」一本一本に付けられた説明タグです。これにより、AIは個々のコードが森全体の中でどのような役割を担っているかを正確に理解します。

単に「何をしているか」ではなく、**「なぜそうしているか」「何を背負っているか」**を記述します。

### タスク管理ツールのクラス設計

これから作成するタスク管理ツールの中核となる`Task`クラスと`TaskManager`クラスを例に、インテント・コメントを記述してみましょう。

**悪い例（Whatしか書いていない）**

```python
# Task class
class Task:
    # initializer
    def __init__(self, id, title):
        self.id = id
        self.title = title
        self.done = False

# TaskManager class
class TaskManager:
    # initializer
    def __init__(self, filepath):
        self.filepath = filepath
        self.tasks = []
```

**良い例（Whyが書かれている）**

```python
/**
 * @class Task
 * @responsibility 個々のタスクの状態（ID, タイトル, 完了状態）を保持するデータ構造。
 * @lifetime TaskManagerによって生成され、ユーザーが削除するまで、あるいはプログラム終了まで生存する。
 * @note このクラスはデータのみを保持し、ビジネスロジック（ファイル保存など）は一切含まないこと。
 */
class Task:
    def __init__(self, id: int, title: str):
        self.id = id
        self.title = title
        self.done = False

/**
 * @class TaskManager
 * @responsibility アプリケーション全体のタスクを管理する。タスクのCRUD操作（生成、読み取り、更新、削除）と永続化（ファイルI/O）に責任を持つ。
 * @relation 集約(Aggregation): 複数のTaskインスタンスをリストとして保持する。Taskのライフサイクル自体には責任を負わない。
 * @singleton このアプリケーションでは、TaskManagerは唯一のインスタンスとして扱われることを想定する。
 * @constraint P1, P3, AC1, AC2 に準拠すること。
 */
class TaskManager:
    def __init__(self, filepath: str):
        self.filepath = filepath
        self.tasks: list[Task] = []
```

**【演習】**: 上記の`TaskManager`に、タスクを追加する`add_task`メソッドをAIと一緒に実装してみましょう。その際、「このメソッドの責務と、AC2の制約を考慮した実装案を教えて」と、**意図を伝えながら**AIに依頼してみてください。

## Step 3: AIエージェントとの対話的開発

グローバル指示書とインテント・コメントという「共通言語」を手に入れた今、いよいよAIとの対話的開発に移ります。

### 1. UMLクラス図の生成

`Step 2`で作成した2つのクラスのコードをAIに渡し、以下のプロンプトで依頼してみましょう。

**プロンプト例**
> 以下のPythonコードと、コード内に記述されたインテント・コメント（特に`@relation`や`@responsibility`）を深く理解した上で、PlantUML形式のUMLクラス図を生成してください。クラス間の関係性（集約など）も正確に表現してください。

AIが`@relation`の記述を元に、単なる關連ではなく「集約（Aggregation）」として図を生成できれば、AIが意図を理解した証拠です。

### 2. 仕様変更への対応

ここで、新しい要求「**タスクを完了する機能を追加したい**」が発生したとします。

**悪い対話例**
> 「タスクを完了する機能を追加して」

これでは、AIはどのクラスにどんな変更を加えるべきか、既存の設計原則を考慮せずに実装してしまうかもしれません。

**良い対話例**
> `TaskManager`クラスに、指定されたIDのタスクを完了状態（`done = True`）にする`complete_task`メソッドを追加してください。
>
> **考慮事項:**
> - 存在しないIDが指定された場合のことを考えて、エラーハンドリングもお願いします。
> - この変更は、我々の設計原則 `P3: 状態の分離` に違反していないか確認してください。
> - 変更後、速やかにファイルに永続化する必要がありますが、`AC2: アトミックな書き込み` の制約を必ず守ってください。

このように、**守ってほしいルール（意図）を明確に伝えながら**開発を進めるのが、AIとの協調の鍵です。

## Step 4: 複数AIとの協調開発 (CIP-Bridge による組織戦)

プロジェクトが大規模になると、一人のAI（副操縦士）だけではコンテキストがパンクします。そこで `CIP-Bridge` を使い、複数の専門家AI（Worker）を束ねて開発を進めます。ここでのあなたは、**「艦隊を指揮する提督」**のような役割になります。

### 1. 人間（機長）が使えるコマンド
Bridgeのターミナル（プロンプト）で以下のコマンドを入力することで、AIの組織を制御・支援できます。

- **`@list`**: 
    - **いつ使うか**: 開発開始時。
    - **効果**: 現在認識されている Worker (ディレクトリ) を一覧表示し、Leader AI に現在の「布陣」を伝えます。AIはこれを見て、適切な Worker に仕事を振るようになります。
- **`@cip [指示内容]`**: 
    - **いつ使うか**: 具体的な実装の合意形成をAI同士に任せたい時。
    - **効果**: **AUTO（自律交渉）モード**を開始します。AI同士が `ARCHITECTURE_MANIFEST.md` に基づいて相談を開始し、結論が出るまで自動的に対話が進みます。
    - **中断方法**: `CTRL+C` を押すと自律モードを抜け、人間に制御権が戻ります。

### 2. システムによる自動サポート機能
`CIP-Bridge` は、人間が「意思決定」に集中できるよう、以下のルーチンワークを自動化しています。

- **ツール実行の自動承認**: 
    - AIがファイルを読み書きする際の「実行許可 [Y/n]」をBridgeが自動的に `ENTER` でパスします。これにより、自律モード中に人間が画面に張り付いている必要がなくなります。
- **組織の自動発見 (Fractal Discovery)**: 
    - 下位ディレクトリで `bridge.py` を起動するだけで、自動的に上位の通信網（`cip_bus`）を見つけ出し、接続されます。

### 3. 実践のコツ：AI間の「衝突」を歓迎する
`CIP-Bridge` 上でAI同士が論争（`[CONFLICT]`）を始めたら、それは設計の曖昧さが浮き彫りになった証拠であり、**「成功」**です。

無理にAI同士で解決させず、人間が介入して「今回のプロジェクトでは A の方針を優先する」と `ARCHITECTURE_MANIFEST.md` を更新してあげることで、健全なソフトウェアが育ちます。

## おわりに

このドキュメントで示したアプローチは、AIを単なるコード生成器ではなく、プロジェクトのルールを理解し、一貫性を保つためのパートナーとして扱うための第一歩です。

これからのエンジニアの価値は、単にコードを書く速さだけではありません。**「いかに的確に、人間とAIの双方に『意図』を伝え、プロジェクト全体を正しい方向に導けるか」**という、アーキテクトやPMに近い能力が、ますます重要になっていくでしょう。
